\section{Implementation} \label{sec:implementation}

The implementation of the certified proof checker consists of three main components:
\begin{itemize}
    \item \textbf{Certificate Parser.} A Haskell program that reads Marabou proof certificates in JSON and generates Rocq source files containing equivalent Gallina definitions.
    \item \textbf{Proof Checker.} A Rocq program that verifies whether a parsed proof certificate constitutes a valid proof of unsatisfiability.
    \item \textbf{Proof Checker Soundness.} A formal proof in Rocq establishing that if the proof checker accepts a certificate, then the original verification query is genuinely unsatisfiable.
\end{itemize}

\autoref{fig:system-workflow} provides an overview of the workflow of the developed system. When Marabou determines that a verification query is UNSAT, it produces a proof certificate of the result in JSON format. The certificate parser uses the resulting JSON proof certificate to generate Rocq source files containing Gallina definitions equivalent to the certificate. The generated files are then imported by the proof checker where it verifies the certificate. Finally, the soundness theorem establishes that any certificate accepted by the proof checker constitutes a valid UNSAT proof of the corresponding verification result.

\subsection{Certificate Parser}
To reason about the proof certificates produced by Marabou in the UNSAT case, they must first be formalised, parsed and then imported into the proof checker.

\subsubsection{Formalisation}
The proof certificate data produced by Marabou is captured by the following Gallina types which exploit the benefits of dependent types.

\begin{itemize}
    \item \texttt{n : nat} --- the number of variables in $x$, as in \autoref{sec:queries}.

    \item \texttt{m : nat} --- the number of equations in \autoref{eq:weighted-sum} and \autoref{eq:relu-auxiliary}, as in  \autoref{sec:queries}.

    \item \texttt{lb, ub : 'rV[R]\_n} --- row vectors of length $n$ representing the lower and upper bounds on each variable.

    \item \texttt{constraint : ('I\_n * 'I\_n * 'I\_n)} --- each ReLU constraint is a triple of dependent indices $(b, f, aux)$ representing the basic, forward, and auxiliary variables.

    \item \texttt{tableau : m.-tuple ('rV[R]\_n)} --- a tuple of $m$ row vectors encoding the linear relationships maintained by Marabou's simplex solver.

    \item \texttt{proof tree : ProofTree.t} --- formalised as an inductive type (\autoref{code:proof-tree})with two constructors: \texttt{node}, which carries split information together with left and right subtrees, and \texttt{leaf}, which carries a contradiction vector.
\begin{lstlisting}[language=Coq, caption={Marabou proof tree as an inductive data type in Rocq.}, label={code:proof-tree}]
Inductive t : Type :=
  (*  split information, left child, right child *)
  | node : Split.t -> t -> t -> t
  (* contradiction vector *)
  | leaf : m-tuple R -> t.
\end{lstlisting}

    \item \texttt{split : Split.t} --- the split information is itself an inductive type (\autoref{code:formalisation-split}), distinguishing single-variable bound tightenings from ReLU phase splits:
\begin{lstlisting}[language=Coq, caption={Split inductive type for case splits.}, label={code:formalisation-split}]
Inductive t : Type :=
  (* variable, value *)
  | single : 'I_n -> R -> t
  (* participating relu variables: b, f, aux *)
  | relu : 'I_n -> 'I_n -> 'I_n -> t.
\end{lstlisting}
\end{itemize}

\subsubsection{Parser Implementation}

In the original Imandra development, this is achieved within the same program: since Imandra is embedded in a subset of OCaml, it can leverage OCaml parsing libraries to read and translate the proof certificates directly at execution time. Consequently, no external preprocessing stage is required.

This approach is not directly transferable to Rocq. The proof checker developed in this work is implemented in Gallina, which cannot invoke arbitrary OCaml parsing code during proof checking. Therefore, a separate preprocessing stage was introduced. A Haskell program parses the JSON proof certificates produced by Marabou and generates Rocq source files containing equivalent Gallina definitions. These generated files are then imported into the proof checker, allowing it to reason about the certificates using native Rocq data structures.

Marabou's proof certificates are not organised in a way that is convenient for recursive verification. In particular, the tightening information is associated with parent nodes, whereas the proof checker benefits from a representation in which this information is readily available when processing each proof step. Consequently, before verification, the parsed certificates should be translated into data types better suited to the recursive structure of the proof checker.

In the original Imandra development, the transformation is carried out within the Imandra code after the certificates have been parsed. In this work, the transformation is instead performed by the Haskell parser while generating the Rocq files. As a result, the Rocq implementation operates directly on the representation expected by the proof checker, keeping the Gallina code focused solely on certificate verification.

\autoref{fig:cert-snippets} shows the correspondence between the JSON certificate format and the generated Gallina code. The first tableau row encodes how each basic variable is expressed as a linear combination of non-basic variables; upper and lower bounds assign intervals to each variable; constraints encode the ReLU relationships; and the proof tree captures the case splits and Farkas witnesses. The Gallina representation mirrors this structure exactly, with real numbers represented as rational fractions and variables indexed by dependent types.

The parser translates each JSON field directly into Gallina definitions: bounds become row vectors of rational numbers using \texttt{\%:R} notation, constraints become tuples of dependent indices via \texttt{inord}, and the recursive proof tree structure maps to nested \texttt{ProofTree.node} and \texttt{ProofTree.leaf} constructors.

\subsection{Proof Checker}


The proof checker has exactly one job: to determine whether an UNSAT proof certificate genuinely certifies unsatisfiability. To do so, it operates over the following data, all of which is obtained from the certificate parser in a form of Gallina files.


As mentioned earlier, the Marabou DNN verifier produces proof certificates in a tree-like structure that the proof checker traverses recursively. \autoref{code:proof-tree} shows the formalisation of Marabou proof trees as an inductive data type. It consists of two constructors: a \texttt{node} and a \texttt{leaf}. Each leaf holds the Farkas contradiction vector, while each node holds the split information required to tighten the bounds, together with references to the left and right child subtrees.


The \texttt{check\_tree} function, shown in \autoref{code:check-tree}, recursively traverses the populated proof tree. At each leaf, the proof checker verifies that the provided Farkas coefficients constitute a valid witness of unsatisfiability for the linear constraints under the current bounds. At each internal node, it first tightens the bounds and then checks that the current split is consistent with the constraints before recursively verifying both child subtrees.

\begin{lstlisting}[language=Coq, caption={Proof checker tree verifier.}, label={code:check-tree}]
Fixpoint check_tree
  (tableau : system m n)
  (ub lb : 'rV[R]_n)
  (constraints : seq Constraint.t)
  (proof_node : ProofTree.t)
  : bool :=
  match proof_node with
  | ProofTree.leaf contradiction =>
    check_contradiction contradiction tableau ub lb
  | ProofTree.node split tleft tright =>
    let bounds := Split.update_bounds_from_split ub lb split in
    let valid_split := Split.check_split split constraints in
    let valid_left  := check_tree tableau bounds.1.2 bounds.1.1 constraints tleft in
    let valid_right := check_tree tableau bounds.2.2 bounds.2.1 constraints tright in
    valid_split && valid_left && valid_right
  end.
\end{lstlisting}
